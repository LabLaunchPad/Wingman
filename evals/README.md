# Wingman Eval Harness

Wingman's "code" is markdown prompts (commands, agents, skills). A structural validator (`plugins/wingman/scripts/validate-structure.mjs`) can check that files exist and frontmatter is well-formed, but it can't tell you whether a skill's *instructions* actually produce the right behavior — that requires running them. This directory is a lightweight behavioral eval harness for exactly that, scoped down from the fuller Tier 1/2/3 pattern studied in `addyosmani-agent-skills` (see `docs/ARCHITECTURE.md` §9) to what Wingman actually needs today.

## How it works

1. **Fixture** (`fixtures/*.sh`) — a script that creates a throwaway, realistic test project with deliberate signals (e.g. a Prisma schema to trigger the Data department, a Dockerfile to trigger DevOps). Wipes and recreates its target directory every run.
2. **Case** (`cases/*.md`) — documents the fixture used, the exact procedure for running the eval, the expected outcome per checkable item, and a **run log** of actual results from real executions — not just what's expected, but what actually happened, verified independently rather than trusted from the tested agent's own self-report.
3. **Run** — spawn a fresh subagent (via the `Agent` tool) that reads only the skill/command under test and the fixture, deliberately not told the expected answer. Grade its actual output against the case's expectations by inspecting the real file tree/content afterward, never by trusting its self-report alone.

## How this splits between CI and manual grading

The harness runs in two tiers, mirroring the gate/periodic split `gstack` uses (see `.github/workflows/`):

- **Deterministic tier — every PR, no API key.** `node scripts/check-fixtures.mjs` (wired into `.github/workflows/validate.yml`) runs every `fixtures/setup-*.sh` into a throwaway dir and asserts it executes cleanly and produces a real git project. This catches a *broken fixture* on every PR, cheaply. `evals/run-headless.mjs --dry-run` additionally confirms every case references a fixture that exists.
- **Behavioral tier — weekly + on-demand, needs `ANTHROPIC_API_KEY`.** `.github/workflows/evals.yml` (`schedule` + `workflow_dispatch`, never per-PR to control token cost) drives a subset of cases through `claude -p` headless via `evals/run-headless.mjs` and uploads the transcripts as an artifact. It **captures**, it does not **grade** — grading stays human/independent, because judging these behavioral cases reliably still needs a real check against the filesystem, not a self-report (this project's core discipline). With no key, the runner degrades to the dry-run integrity check above.

So the *stochastic, model-in-the-loop* half is run on a schedule but graded by a human reading the run log; the *deterministic* half gates every PR. The run log in each case file remains the record of when a case was last actually graded, so staleness stays visible.

## Trust levels

Borrowed from `addyosmani-agent-skills`' eval schema:
- **`provisional`** — passed at least one real run, single scenario, manually graded. Most Wingman eval cases will sit here.
- **`verified`** — passed multiple differently-shaped scenarios, including at least one negative case (confirming the skill correctly does *nothing* when it shouldn't act).

## Existing cases

**As of 2026-07-10, all 12 cases are `verified`.** The original 10 each passed a second, differently-shaped scenario (a negative, inverse, or adversarial case), independently checked against the real filesystem; the per-case notes below summarize how each *first* passed, and every case doc's run log carries the second-run detail. The single most notable second run: `full-pipeline-e2e`'s adversarial fixture produced the first real-independent-dispatch `DO NOT SHIP` (rejecting an already-committed diff with SQL injection + plaintext passwords) — the one thing two "realistic" runs couldn't. Two new cases (`boardroom-deep-review`, `drift`) were added for the v8 four-command extension and each passed `verified` on their first run, per the raised eval bar the founder's own Boardroom review set for that work.

- `cases/department-lead-activation.md` — tests `skills/department-lead-activation`. `verified`: passed a positive case (all conditional signals present), a negative case (none present, confirming no over-triggering), and a third scenario (v8) covering the tech-stack/MCP skill-catalog materialization step — exactly one narrowly-scoped skill file off a real signal, correctly left alone on a second run.
- `cases/evolve-promotion.md` — tests `skills/evolve-promotion`. `verified`: passed a positive case (gather/cluster/classify/propose, then file placement under a simulated approval — the real `AskUserQuestion` gate can't be exercised by a background subagent with no real founder to answer it) and a negative case that specifically tests same-incident-double-logging isn't mistaken for genuine repetition.
- `cases/full-pipeline-e2e.md` — tests the whole SDLC pipeline (`plan` → `build` → `secure` → `ship`) run in sequence against one continuous realistic project, not one skill at a time. First pass caught and fixed one real process defect (a build-stage Boardroom checkpoint that was never actually recorded despite all 5 seats reporting — see the fix in `commands/boardroom.md`); now `verified` after a Run 3 adversarial fixture (an already-committed SQL-injection-plus-plaintext-passwords diff) produced the first real-dispatch `DO NOT SHIP`.
- `cases/hotfix.md` — tests `commands/hotfix.md`'s root-cause-first discipline end to end against a real, subtle bug. `provisional`: passed a single-scenario run (an unambiguous logic bug); correctly reproduced before fixing, wrote a genuine red-then-green regression test, and recorded the checkpoint correctly.
- `cases/launch.md` — tests `commands/launch.md`'s self-activating Growth signal, scope discipline, and the Boardroom's "content passed directly" review path (the one scope type no other eval exercises). `provisional`: passed a single-scenario run; surfaced and fixed a real gap in `boardroom.md`'s `scope_ref` schema for directly-passed content.
- `cases/boardroom-gate-rule.md` — tests `commands/boardroom.md`'s consolidation logic directly, using synthetic seat verdicts rather than a real review, specifically covering the `DO NOT SHIP` path no other eval had ever hit. `provisional`: confirmed the rule is an unambiguous "any NO_GO" predicate; surfaced and fixed a real gap in what `next_stage` should be on a blocked checkpoint (previously undefined, risked silently advancing a blocked project past its gate).
- `cases/systematic-auditing.md` — tests `skills/systematic-auditing` + `commands/audit.md` against a fixture with 3 deliberately differently-shaped seeded issues (config/wiring, doc-drift, test-coverage gap). `provisional`: found all 3 plus 2 legitimate bonus issues, zero false positives, fixed everything contained and correctly deferred one genuine product decision. Also confirmed nested subagent dispatch (a `general-purpose` agent spawning further `general-purpose` agents) works in this sandbox, distinct from the separately-established constraint that *named custom* `subagent_type`s don't.
- `cases/verification-before-completion.md` — tests the single most load-bearing discipline skill in the project, against a scenario deliberately engineered so the lazy answer is wrong (a "fixed" bug whose fix actually introduces a regression). `provisional`: correctly ran real verification instead of judging the diff by eye, and reported the accurate, more nuanced result.
- `cases/retro.md` — tests `commands/retro.md` against a project with genuinely rich real history (reusing `full-pipeline-e2e` Run 2's fixture post-ship). `verified`: first pass gathered real facts, wrote a specific (not generic) retro in the exact required format, and made a reasoned rather than reflexive feed-forward decision about `/wingman:learn`; a second, differently-shaped scenario in its run log promoted it.
- `cases/harness.md` — tests `commands/harness.md`'s 5-point checklist against two differently-shaped subjects: Wingman's own real repo (positive — correctly identified missing CI as the one genuine gap, leading to a real fix, `.github/workflows/validate.yml`) and a fixture with a rubber-stamp test suite (negative — correctly refused to trust a green "4 passing" line, named the exact fake mechanism, and performed a real spot-check). `verified` on first run, since the two scenarios were differently shaped from the start.
- `cases/boardroom-deep-review.md` (v8) — tests `commands/boardroom.md`'s opt-in deep-review mode with a real 2-round, 10-subagent dispatch (not synthetic verdicts). `verified` on first run: a genuinely cross-informed verdict revision (Cost seat, `GO_WITH_CONCERNS` → `NO_GO`, explicitly citing Security's round-1 finding), the convergence check correctly stopping at round 2 rather than over-triggering a round 3, and a diff check confirming the default no-flag path is byte-for-byte unaffected.
- `cases/drift.md` (v8) — tests `commands/drift.md`'s detect/route/capture logic across 3 scenarios: interactive-positive, negative (already-covered), and headless-fallback. `verified` on first run — and found two real gaps in the process, both fixed before promotion: a `boardroom-checkpoint.mjs` bug where a stale approval could wrongly clear an unreviewed amendment (fixed: the hook now only trusts a checkpoint that's genuinely the last thing in the file), and an underspecified `checkpoints.jsonl` field convention for the no-dispatch headless case (fixed: pinned the exact shape in `drift.md`, documented the one-time `bottom_line` enum exception in `docs/DATABASE.md`).

**Two lighter, targeted checks (not full cases — no dedicated fixture script or case doc, per the eval-commitments bar the founder's own review set for v8):**
- **`/wingman:agents` dry-run** against a fixture with a fully populated `.wingman/state.json` (5 department leads, 2 specialists) and real `.claude/agents/*.md` files for each. First attempt surfaced a real gap — the command had no instruction for *where* to source each agent's one-line description, and a test subagent reached for `docs/ARCHITECTURE.md`/`docs/AGENT-ROSTER.md` (Wingman-repo-only docs that don't ship with the installed plugin) — the same "docs/ isn't installed, so a shipped file can't depend on it" bug class this project has hit more than once before (see `docs/PROJECT.md`'s decisions log). Fixed: `agents.md` now explicitly reads each agent's own `.claude/agents/<name>.md` file in the founder's project and is told never to fall back to Wingman's own non-shipped docs. Re-verified clean on a second run.
- **`plan.md`'s amendment mode**, byte-for-byte append-only check — covered directly by `cases/drift.md`'s Scenario 1 (its run log verifies the original file's content hash is unchanged across the edit and only additive lines appear after it), not duplicated as a separate case.
