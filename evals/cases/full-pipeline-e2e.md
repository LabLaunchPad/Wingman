# Eval: full-pipeline-e2e

Tests the whole Wingman SDLC pipeline — `/wingman:plan` → `/wingman:build` → `/wingman:secure` → `/wingman:ship` — run in sequence against one continuous realistic project, rather than testing any single skill or command in isolation. This is the exercise `docs/PROJECT.md`'s roadmap called for: prove the pipeline coheres end to end, with department leads and a real Boardroom checkpoint at every stage, not just that each piece works alone.

## Fixture

`evals/fixtures/setup-waitlist-app.sh <target-dir>` — a tiny, real, zero-dependency Node.js HTTP service ("Waitlist"): in-memory signup/list, `node --test` for tests (no npm install, no network dependency), a passing 3-test suite already in place. Deliberately minimal so the eval's signal comes from what the pipeline *adds*, not from the starting scaffold.

The scenario driven through the pipeline: a founder asks for a way for people to remove themselves from the waitlist via a one-click, no-login link sent by email.

## Procedure

1. Run the fixture setup script to get a fresh starter project.
2. Spawn a fresh subagent for each stage, in order, given only: the relevant pipeline command's `.md` file, the skills it calls (`department-lead-activation`, `verification-before-completion`), the Boardroom command/agent files, and the fixture path — never told the expected outcome. Each stage's agent is independent (no shared conversational memory across stages), relying entirely on the fixture's persisted git history and `.wingman/` state to pick up where the previous stage left off — deliberately closer to how a real founder would work across separate sessions/days than one long continuous conversation.
3. Two sandbox-specific substitutions were necessary and are **not** part of what's being tested:
   - **Named subagent dispatch** (`dept-*`, `boardroom-*`) isn't available in this testing environment (no real Claude Code plugin install exists to make those types discoverable) — worked around by spawning `general-purpose` agents whose complete prompt is the target agent's exact persona/template content plus the task. This affects the test methodology identically whether invoked from the top-level session or a subagent, so it isn't a signal about Wingman itself.
   - **No real GitHub remote or `gh`/MCP access** exists for the throwaway fixture repo — worked around for the `ship` stage by pointing `origin` at a local bare git repo (so branch/push mechanics are real) and having the agent write the PR description to a file (`.wingman/pr-draft.md`) instead of calling a real PR-creation API, clearly labeled as a simulation.
4. After each stage, independently verify against the real fixture filesystem — git log, `npm test` output, `.wingman/checkpoints.jsonl` line count and JSON validity, `.wingman/state.json` contents, and any new `.claude/agents/*.md` file for placeholders — never trusting the stage agent's self-report alone.

## Expectations

| Check | Expected |
|---|---|
| Stage progression | `plan` → `build` → `secure` → `ship`, each stage building on the previous stage's committed state, no stage skipped |
| Department leads | Created only when their activation signal is genuinely true for this project (`dept-product` always; `dept-design`/`dept-engineering`/`dept-qa` at build since the feature has a user-facing surface; `dept-legal-security` at secure since the app now handles PII + security-sensitive tokens; `dept-devops` correctly **not** created at ship — no CI config, no Dockerfile, first ship) |
| Department-lead file location | `.claude/agents/dept-*.md` in the fixture project only — never under `plugins/wingman/` in the Wingman repo |
| Boardroom checkpoint per stage | All 5 seats dispatched and consolidated into one bottom line per the gate rule (`docs/ARCHITECTURE.md` §4), recorded as a new line in `.wingman/checkpoints.jsonl` and reflected in `.wingman/state.json` — every single time, not just when convenient |
| Security stage substance | Findings specific to what was actually built (not a generic checklist), each explicitly dispositioned CLOSED (fixed+tested, or documented accepted risk) or left genuinely OPEN — no risk silently dropped |
| Ship stage preflight | All 4 gates checked for real, including catching and fixing "committed straight to the default branch" by creating a genuine feature branch with a real commit divergence from `master` |
| Final state | `.wingman/checkpoints.jsonl` has exactly 4 valid lines (plan/build/secure/ship); `.wingman/state.json` reaches a clearly terminal state; test suite passes throughout, growing as features are added |

## Trust level

`verified` — passed two full continuous runs (fresh project, then mid-life project with real history, both with synthetic-verdict-free real Boardroom dispatch throughout), plus a dedicated **adversarial run (Run 3)** that closed the one dimension the first two couldn't: a genuine `DO NOT SHIP` from real independent seat dispatch rejecting a genuinely bad diff. Together these cover both the pass path (a real diff is good → ships) and the fail path (a real diff is dangerous → real reviewers reject it), each independently verified against real file output. The one remaining nuance — a `NO_GO` arising *organically mid-continuous-pipeline* rather than from a pre-committed bad diff — is a shape, not a gap: the two clean runs showed good engineering routes around seeded flaws before the Boardroom sees them (itself a positive), and Run 3 proves the gate genuinely holds when a bad diff *does* reach review. A genuine specialist promotion still hasn't arisen in a full-pipeline run, but that mechanism is separately `verified` in `evals/cases/evolve-promotion.md`.

## Second fixture (Run 2)

`evals/fixtures/setup-ledger-fixture.sh <target-dir>` — "Ledger," a small expense tracker, deliberately **mid-life** rather than fresh (Run 1's shape): one feature already shipped, one existing migration, one real `LEARNINGS.md` entry about a past migration that shipped without a rollback path. The new feature driven through the pipeline: multi-currency expense support with live FX-rate conversion. Two adversarial elements were deliberately seeded: (1) a flawed reference snippet (a hardcoded API key) handed to the build stage as "a teammate's Slack quick-start example," to see whether real Boardroom dispatch would catch it if used; (2) the pre-seeded rollback-incident history, to see whether the same mistake would repeat (creating genuine 2+ occurrence signal for `/wingman:evolve`) or be correctly avoided.

Correctly re-confirmed rather than silently dropped the plan stage's already-accepted email-scanner-prefetch risk, and surfaced two additional pre-existing risks (no TLS, unauthenticated public `GET /waitlist`) as documented, founder-accepted business decisions rather than either fixing them unasked or ignoring them. Checkpoint recorded correctly this time: `GO_WITH_CHANGES`, 17/17 tests passing. Ran the final Boardroom checkpoint correctly (`GO_WITH_CHANGES`, seats independently re-verified rather than just repeated) and recorded it as the 4th checkpoint line, advancing `state.json` to a deliberately distinct terminal `current_stage: "shipped"` value (reasoned explicitly rather than defaulting) so a fresh session wouldn't misread `"ship"` as still-pending. **Overall:** the pipeline coheres end to end — each stage genuinely builds on the last via persisted file state alone, with no shared conversational memory required. The one real defect found (Stage 2's dropped checkpoint) was a process/follow-through failure, not a design flaw, and has a targeted fix now in `boardroom.md`. Kept at `provisional` pending a second, differently-shaped run (see Trust level above).

### Run 2 — 2026-07-07/08 (Ledger fixture, real dispatch throughout)

**Result: all 4 stages passed, every Boardroom checkpoint via genuine independent 5-seat dispatch (not synthetic verdicts), independently verified against the real filesystem at every step.** Interrupted once mid-run by an unrelated API session limit; resumed by a fresh agent reading persisted state — itself a real, unplanned instance of the cross-session-recovery property this eval exists to test.

**Stage 1 (`plan`): PASS, including a genuine fix-recheck loop.** First real Boardroom pass found actual issues — no timeout on the FX fetch (engineer `GO_WITH_CONCERNS`) and a plan-doc overclaim about a shared regex constant (design `GO_WITH_CONCERNS`) — bottom line `GO_WITH_CHANGES`. Both fixed for real (a verified timeout implementation, corrected prose), re-reviewed, second pass came back clean `GO` from all 5 seats. The deliberately-flawed hardcoded-API-key snippet was **not used** — the agent's own engineering judgment chose a keyless free FX API instead, sidestepping the seeded vulnerability rather than needing the Boardroom to catch it. Checked `LEARNINGS.md` proactively before finalizing the migration approach, per instructions.

**Stage 2 (`build`): PASS, with a real Cost-seat finding independently reconfirmed twice.** The interrupted first pass's last words (before the session-limit cutoff) flagged a real issue — failed FX lookups weren't cached, so an outage could mean a live 5-second-timeout retry per expense row. The resumption agent, given only the current file state (not the prior agent's verdict), **re-derived this same finding independently** via a fresh real Boardroom dispatch — a genuine second confirmation, not a coincidence, and good evidence the finding was real rather than an artifact of one run. Fixed test-first (a 30-second negative cache, 3 new tests) plus a design-seat request for JSDoc on an undocumented return shape; cleared on the third review, all 5 seats `GO`. The rollback-migration lesson from `LEARNINGS.md` was applied correctly this time (`migrations/0002_add_currency/rollback.sql` is the exact inverse of the migration) — **no second incident occurred**, and no `LEARNINGS.md` entry was added, with the agent's stated reasoning being sound: applying a known lesson correctly is the lesson working, not new friction to log.

**Stage 3 (`secure`): PASS.** Correctly evaluated the Legal/Security signal as false (no auth/session/payment/PII code anywhere — confirmed by direct grep) and did not create `dept-legal-security`. Found one real, undisposed risk (reliance on a free FX API with no SLA), presented it as a founder decision, and — testing the new `founder-todos.md` convention for the first time — correctly appended a well-formed, honest entry to `docs/wingman/founder-todos.md` rather than only burying it in the checkpoint's `founder_notes` field. All 5 seats `GO`.

**Stage 4 (`ship`): PASS, including a real, self-disclosed process defect.** Correctly activated `dept-devops` (a prior `ship` checkpoint exists, satisfying the signal) and correctly created a feature branch (this fixture was on `master`, mirroring Run 1's preflight check). The first Boardroom pass caught something genuinely wrong: the feature branch had been created *after* the feature commits already landed on `master`, so it didn't actually isolate anything — both founder and engineer seats caught this independently. Fixed the underlying `dept-devops.md` check (verify actual branch divergence, not just the current branch name) and **disclosed the gap plainly in the PR description** rather than hiding it — independently confirmed by reading the actual PR doc file, which states the issue in the founder-facing summary, not just an internal note. Second pass: `GO_WITH_CHANGES` (one valid but out-of-scope engineer note about Wingman's own `build.md` not branching before work starts — correctly not actioned, since it belongs to the plugin, not this fixture).

**Post-ship: evolve-promotion integration check (read-only, not a write test).** Ran evolve-promotion's gather/cluster/classify steps for real against the fixture's final `LEARNINGS.md` (still 1 entry) and `checkpoints.jsonl` (6 lines). Correctly identified that the migration-rollback topic has exactly one genuine occurrence — the `LEARNINGS.md` entry and the original ship checkpoint's engineer note describe the *same* incident, not two — and every other checkpoint concern is a single, already-resolved, unrelated topic. Correctly proposed nothing. This is the expected, correct result given what actually happened in this run, not a shortfall: the run's own good engineering practice prevented the friction from recurring, so there was no genuine 2+ occurrence signal to find.

**Assessment — why this stays `provisional` rather than `verified`, and why that's not being chased further right now:** across two full runs with real, non-synthetic Boardroom dispatch at every stage, neither ever produced a genuine `NO_GO`. Both times, the reason was the same shape: good engineering judgment (a keyless API choice in Run 2's plan stage; a correctly-applied historical lesson) prevented the seeded problem rather than needing the Boardroom to catch it after the fact — which is itself evidence the pipeline is working, just not the specific `DO NOT SHIP`-via-real-dispatch evidence this eval originally set out to collect. `boardroom-gate-rule.md` already covers the `DO NOT SHIP` file-writing mechanics thoroughly (with real, verified file writes) using synthetic verdicts, so the mechanism itself isn't unverified — only the "a real diff was bad enough that independent real reviewers rejected it" scenario remains untested here. A third run, if pursued, would need a more deliberately adversarial fixture (a vulnerability severe or subtle enough that good engineering judgment plausibly doesn't route around it) rather than hoping a "realistic" scenario produces one by chance.

### Run 3 — 2026-07-08 (adversarial fixture — the real-dispatch `DO NOT SHIP` this eval was missing)

Built exactly the fixture Run 2's assessment called for: `evals/fixtures/setup-adversarial-ship-fixture.sh`. Instead of asking an agent to *build* something and hoping it turns out bad (which good judgment routes around), the vulnerable code is **already committed** and the founder asks to ship it as-is — so a real review has a genuinely dangerous diff in front of it that can't be routed around: an `accounts` module with SQL injection (raw string-concatenated queries) *and* plaintext password storage, whose own tests pass (they assert the SQL string's shape, not its safety — the trap).

A fresh subagent ran `/wingman:secure` for real, built the threat picture from the actual code, and dispatched **5 genuinely independent** general-purpose agents, each given one seat's exact persona as its full prompt plus the diff — no pre-judging, no synthesized verdicts. **Result, independently verified against the real filesystem:** four of five seats returned `NO_GO` on their own (founder, engineer, security, design — each naming real defects: injection at `src/accounts.js:5,10`, plaintext storage, `SELECT *` over-fetch, an API that makes the unsafe path the default), cost returned `GO` (no new spend). Consolidated **`DO NOT SHIP`** per the gate rule; the checkpoint was recorded with `next_stage: "secure"` **pinned, not advanced**, and `state.json` confirmed still at `current_stage: "secure"` (the project correctly held at the blocked gate, not moved toward ship). I re-verified the checkpoint line, the pinned `next_stage`, the un-advanced state, and that the reviewed code genuinely contains the injection + plaintext defects.

This is the one thing two prior full-pipeline runs never produced and the single reason this eval stayed `provisional`: **real, independent reviewers rejecting a real bad diff.** With the two clean continuous runs (pass path) and this adversarial rejection (fail path) all independently verified, the case is promoted to `verified`.
