# Eval: full-pipeline-e2e

Tests the whole Wingman SDLC pipeline — `/wingman:plan` → `/wingman:build` → `/wingman:secure` → `/wingman:ship` — run in sequence against one continuous realistic project, rather than testing any single skill or command in isolation. This is the exercise `docs/PROJECT.md`'s roadmap called for: prove the pipeline coheres end to end, with department leads and a real Boardroom checkpoint at every stage, not just that each piece works alone.

## Fixture

`evals/fixtures/setup-waitlist-app.sh <target-dir>` — a tiny, real, zero-dependency Node.js HTTP service ("Waitlist"): in-memory signup/list, `node --test` for tests (no npm install, no network dependency), a passing 3-test suite already in place. Deliberately minimal so the eval's signal comes from what the pipeline *adds*, not from the starting scaffold.

The scenario driven through the pipeline: a founder asks for a way for people to remove themselves from the waitlist via a one-click, no-login link sent by email.

## Procedure

1. Run the fixture setup script to get a fresh starter project.
2. Spawn a fresh subagent for each stage, in order, given only: the relevant pipeline command's `.md` file, the skills it calls (`department-lead-activation`, `verification-before-completion`), the Boardroom command/agent files, and the fixture path — never told the expected outcome. Each stage's agent is independent (no shared conversational memory across stages), relying entirely on the fixture's persisted git history and `.wingman/` state to pick up where the previous stage left off — deliberately closer to how a real founder would work across separate sessions/days than one long continuous conversation.
3. Two sandbox-specific substitutions were necessary and are **not** part of what's being tested:
   - **Named subagent dispatch** (`dept-*`, `boardroom-*`) isn't available in this testing environment (no real Claude Code plugin install exists to make those types discoverable) — worked around by spawning `general-purpose` agents whose complete prompt is the target agent's exact persona/template content plus the task. This affects the test methodology identically whether invoked from the top-level session or a subagent, so it isn't a signal about Wingman itself.
   - **No real GitHub remote or `gh`/MCP access** exists for the throwaway fixture repo — worked around for the `ship` stage by pointing `origin` at a local bare git repo (so branch/push mechanics are real) and having the agent write the PR description to a file (`.wingman/pr-draft.md`) instead of calling a real PR-creation API, clearly labeled as a simulation.
4. After each stage, independently verify against the real fixture filesystem — git log, `npm test` output, `.wingman/checkpoints.jsonl` line count and JSON validity, `.wingman/state.json` contents, and any new `.claude/agents/*.md` file for placeholders — never trusting the stage agent's self-report alone.

## Expectations

| Check | Expected |
|---|---|
| Stage progression | `plan` → `build` → `secure` → `ship`, each stage building on the previous stage's committed state, no stage skipped |
| Department leads | Created only when their activation signal is genuinely true for this project (`dept-product` always; `dept-design`/`dept-engineering`/`dept-qa` at build since the feature has a user-facing surface; `dept-legal-security` at secure since the app now handles PII + security-sensitive tokens; `dept-devops` correctly **not** created at ship — no CI config, no Dockerfile, first ship) |
| Department-lead file location | `.claude/agents/dept-*.md` in the fixture project only — never under `plugins/wingman/` in the Wingman repo |
| Boardroom checkpoint per stage | All 5 seats dispatched and consolidated into one bottom line per the gate rule (`docs/ARCHITECTURE.md` §4), recorded as a new line in `.wingman/checkpoints.jsonl` and reflected in `.wingman/state.json` — every single time, not just when convenient |
| Security stage substance | Findings specific to what was actually built (not a generic checklist), each explicitly dispositioned CLOSED (fixed+tested, or documented accepted risk) or left genuinely OPEN — no risk silently dropped |
| Ship stage preflight | All 4 gates checked for real, including catching and fixing "committed straight to the default branch" by creating a genuine feature branch with a real commit divergence from `master` |
| Final state | `.wingman/checkpoints.jsonl` has exactly 4 valid lines (plan/build/secure/ship); `.wingman/state.json` reaches a clearly terminal state; test suite passes throughout, growing as features are added |

## Trust level

`provisional` — passed one full run, single scenario, manually and independently graded at every stage. Has not yet been run against a second, differently-shaped scenario (e.g. a project where a signal should read as *false* partway through, or one that needs a real `NO_GO`/`DO NOT SHIP` outcome) — that would be needed to promote this to `verified`, matching the bar the other two eval cases already met.

## Run log

### Run 1 — 2026-07-07

**Stage 1 (`plan`): PASS.** Produced a real plan file (`docs/wingman/plans/2026-07-07-waitlist-self-removal.md`) covering removal tokens, an unsubscribe route, and a mailer stub, with a Plain-Language Summary that correctly flagged the gap between "confirmation email" (asked for) and "logged, not sent" (what's actually cheap to build without a new paid dependency) rather than silently overpromising. Boardroom checkpoint recorded: all 5 seats `GO_WITH_CONCERNS` except cost `GO`, bottom line `GO_WITH_CHANGES`, `founder_decision: ship_it`.

**Stage 2 (`build`): PASS on implementation, FAIL on process — caught and fixed.** The removal-token mechanism, unsubscribe route, and mailer stub were implemented correctly and test-first (15/15 passing), and `dept-design`/`dept-engineering`/`dept-qa` were created correctly in the fixture project. However, independent verification found the stage's Boardroom checkpoint was **never actually recorded** — `.wingman/checkpoints.jsonl` still showed only 1 line (the plan checkpoint) and `.wingman/state.json` was stale, despite all 5 seats having genuinely reported (founder/engineer/security/cost `GO`, design `GO_WITH_CONCERNS` on bare unsubscribe-page HTML lacking a `<title>`/back-link — independently corroborated by reading the actual generated HTML in `src/server.js`). This is a real finding about the orchestrating agent's follow-through on a multi-step process, not a wording gap in `boardroom.md` (its "Record the checkpoint" section already said "always, regardless of scope"). The checkpoint was manually consolidated and written using the seats' actual (verified) verdicts, since re-running the whole stage would have discarded genuine, already-corroborated review work. **Fix applied to `boardroom.md`**: added an explicit instruction to re-read both `.wingman/` files from disk before reporting a checkpoint as done, tying the step to the `verification-before-completion` skill by name — directly targeting this failure mode (a stage narrating completion without confirming the write actually landed).

**Stage 3 (`secure`): PASS.** Correctly identified the Legal/Security activation signal as true (PII storage + security-sensitive tokens) and created `dept-legal-security` with no unfilled placeholders. Found two genuine, previously-undetected vulnerabilities specific to this codebase — cleartext removal-token logging and unescaped-newline log injection in `src/mailer.js` — and fixed both test-first (regression tests that capture real `console.log` output, not just call the function). Correctly re-confirmed rather than silently dropped the plan stage's already-accepted email-scanner-prefetch risk, and surfaced two additional pre-existing risks (no TLS, unauthenticated public `GET /waitlist`) as documented, founder-accepted business decisions rather than either fixing them unasked or ignoring them. Checkpoint recorded correctly this time: `GO_WITH_CHANGES`, 17/17 tests passing. Independently verified: 3 valid JSON lines in `checkpoints.jsonl`, `state.json` advanced correctly, no placeholders in the new agent file.

**Stage 4 (`ship`): PASS.** Correctly evaluated the DevOps activation signal as false (no CI config, no Dockerfile, first ship) and did not create `dept-devops` speculatively. Caught the "committed straight to default branch" preflight failure for real (the fixture had been on `master` throughout) and fixed it properly — created `ship/self-removal` off `master`, pushed it to the local bare origin, leaving `master` on origin genuinely unmoved (verified via direct inspection of the bare repo, not just the fixture's local view). Produced a well-formed, clearly-labeled simulated PR draft (title, plain-language summary, technical summary, test plan, checkpoint history) since no real GitHub remote existed. Ran the final Boardroom checkpoint correctly (`GO_WITH_CHANGES`, seats independently re-verified rather than just repeated) and recorded it as the 4th checkpoint line, advancing `state.json` to a deliberately distinct terminal `current_stage: "shipped"` value (reasoned explicitly rather than defaulting) so a fresh session wouldn't misread `"ship"` as still-pending. Did not run `launch`/`retro`/`learn`/`telemetry`, per instructions. Independently verified: 4 valid checkpoint lines, `git ls-remote`/bare-repo inspection confirming genuine branch divergence, 17/17 tests, and no writes leaked into `plugins/wingman/` in the Wingman repo itself.

**Overall:** the pipeline coheres end to end — each stage genuinely builds on the last via persisted file state alone, with no shared conversational memory required. The one real defect found (Stage 2's dropped checkpoint) was a process/follow-through failure, not a design flaw, and has a targeted fix now in `boardroom.md`. Kept at `provisional` pending a second, differently-shaped run (see Trust level above).
