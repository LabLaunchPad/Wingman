## 2026-07-23 - [Test-Presence Content Scanning Optimization]
**Learning:** In multi-stage push gates or validation workflows, checking changed files against test coverage via content scanning (checking if any test file imports/requires a source file) is extremely slow if implemented naively. Doing recursive directory walking and file reading inside a loop over M changed files results in O(M * N) synchronous I/O operations (where N is the number of test files). Lazily caching the test files list and their contents exactly once inside the entry gate function isolates state, ensures 100% test concurrency safety, and drops I/O operations to at most O(N), saving hundreds of milliseconds on large change-sets.
**Action:** When performing content-based scanning of multiple files against a target set of resources, always lazily cache the target resources (and their contents) exactly once at the entry function level instead of re-reading them recursively from disk.

## 2026-07-22 - [Check-Fixtures Async I/O]
**Learning:** Running multiple parallel processes (like 49 setup-fixture bash scripts via an async worker pool in Node.js) while making blocking synchronous file-system calls (`readFileSync`, `readdirSync`, `mkdtempSync`, `rmSync`) inside each task blocks the single-threaded event loop. This degrades parallel execution performance and process management latency. Replacing them with non-blocking async promises (`node:fs/promises` APIs) unblocks the event loop completely, optimizing signal detection and file manipulation during heavy parallel task executions.
**Action:** Always prefer promise-based async I/O (`fs/promises`) inside any concurrent or parallel process execution loop (such as child process execution pools) in Node.js to keep the event loop highly responsive.
