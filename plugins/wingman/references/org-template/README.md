# Org-template reference

Three pieces of static guidance content, cited from the pipeline and from the `memory` skill, for
context a founder builds up *across* projects rather than within a single one. This is deliberately
narrow — see `docs/PROJECT.md`'s decisions log (2026-07-22, "founder-org scaffold") for the fuller
"master repo blueprint" this was scoped down from, and why the rest of it wasn't built.

- `project-types/` — a catalog + 7 short playbooks for the kinds of projects a solo founder
  typically builds, consulted by `commands/pipeline/discovery.md` when a project's shape clearly
  matches one.
- `founder-preferences.md` — guidance on what's worth learning about a founder's working style,
  approval preferences, and output/stack defaults, and where to persist it once known (the `memory`
  skill's `.wingman/memory/MEMORY.md`, not a new store).
- `capability-map.md` — guidance on what a founder's own capability/skill-gap picture covers, and
  how to use it to calibrate explanation depth.

**None of this is a data-capture mechanism.** There is no command that instantiates these into a
founder's project and no generation step — they are read-only reference content an existing
command/skill consults, the same way `references/testing-patterns.md` or
`references/security-checklist.md` already work. If real per-founder, cross-project capture is ever
needed, that is a separate, future evidence-gated decision (see `skills/evidence-gated-catalog`).
