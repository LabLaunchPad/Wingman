# Freelancing delivery

**Shape:** building for a paying client's business, not the founder's own product.

**What changes in the pipeline:**
- Discovery: the "founder" in the room and the actual end-user may be different people — clarify
  whose success signal matters (the client's, their customers', or both) before scoping.
- Governance: the client, not the builder, usually owns final ship/no-ship authority — Boardroom
  checkpoints should surface this explicitly rather than assume the person running Wingman decides.
- Definition of Done: contract/scope-of-work terms are a real constraint the Build stage's DoD gate
  should reflect — don't let `engineering-minimalism` trim something the client explicitly paid for.
- Handoff: plan for a clean handoff (docs, credentials transfer, no dependency on the builder's own
  tooling continuing to exist) as part of Ship, not an afterthought.

**Common over-scoping trap:** building beyond the agreed scope because it seems like "better
engineering" — extra scope on someone else's dime is still scope creep.
