## Decisions log

Durable decisions only — not every turn-level choice. Newest first.

- **Phase 2's three new evals each surfaced a real, previously-undocumented gap in `boardroom.md`, none of which the Phase-1 fixes or the original e2e run had caught.** Built purpose-made, minimal, deterministic fixtures for each (a discount calculator with a floor-vs-round bug for `hotfix`; a pre-shipped feature for `launch`; a trivial one-line diff for testing `boardroom.md`'s consolidation logic directly with synthetic seat verdicts, to isolate the rule from seat-judgment quality). Findings: (1) `next_stage`'s meaning on a `DO NOT SHIP` checkpoint was undefined — since `state.json`'s `current_stage` is set from `next_stage` unconditionally, a literal reading of the schema's other (advancing) examples could have let a blocked checkpoint silently advance the project past its own gate; now explicit that `next_stage` stays pinned to the reviewed stage on `DO NOT SHIP`. (2) `scope_ref`'s format for content passed directly by a calling command (launch copy, not a plan or diff) was undocumented, forcing each caller to improvise; now a documented convention. Neither gap could have been found by re-running `full-pipeline-e2e` again, since that eval's single scenario never produced a `DO NOT SHIP` outcome or exercised the directly-passed-content path more than once. Token-efficiency choice made explicitly: declined to build a separate dedicated eval for the 5 Boardroom personas, since they'd already been dispatched ~20+ times incidentally across other evals with their output contract holding every time — spending a fresh eval on them would mostly re-confirm existing evidence.
- **A four-way parallel expert audit (architecture, content quality, executable code, eval coverage) found and fixed 5 real bugs no single pass had caught.** Run 2026-07-07 as a deliberate cross-check after the e2e eval, each auditor scoped to a distinct concern and given exact file lists to stay token-efficient. Findings, most severe first: (1) `hooks/hooks.json` registered the plan-mode safety gate under `PermissionRequest`, which is not a real Claude Code hook event (the valid set is `SessionStart`/`SessionEnd`/`UserPromptSubmit`/`Stop`/`StopFailure`/`PreToolUse`/`PostToolUse`) — confirmed against the authoritative hooks reference, meaning the entire `boardroom-checkpoint` enforcement had been silently inert since it was built; fixed to `PreToolUse` with the `ExitPlanMode` matcher, and `boardroom-checkpoint.mjs`'s `allow()` output corrected to the real `hookSpecificOutput.permissionDecision` field, plus its malformed-stdin path changed from fail-open to fail-closed to match the "deterministic backstop" intent it was already documented as having. (2) `boardroom.md`'s "Record the checkpoint" step both cited non-shipped `docs/DATABASE.md` for `state.json`'s exact shape (the same runtime-dependency-on-a-non-shipped-doc bug class as the earlier `evolve-promotion`/`department-lead-activation` fixes) and never instructed preserving `active_department_leads`/`active_specialists` when overwriting `state.json` — literally followed, every checkpoint would have silently wiped the department-lead/specialist roster. Fixed: the full schema is now inlined in `boardroom.md`, and the step is explicitly a read-merge-write, not a blind overwrite. (3) `skills/department-lead-activation/SKILL.md` never actually wrote to `state.json` at all, despite `docs/DATABASE.md` and `evolve-promotion` both asserting it populates `active_department_leads` — fixed by adding that step explicitly. (4) `boardroom-engineer.md`/`boardroom-security.md` both declared `model: inherit` despite `docs/ARCHITECTURE.md` §8 explicitly assigning them `model: opus` as the two highest-stakes seats — fixed. (5) Several doc-drift items (`CLAUDE.md` omitted `launch`/`hotfix` from its adaptive-commands list; `ARCHITECTURE.md`'s launch row and "Open items"/rollout sections were stale relative to what's actually built and tested; `ship.md`'s final checkpoint was conditional on "meaningful release," contradicting the stated one-checkpoint-per-stage design) — all fixed. General lesson: none of these would have been caught by re-running the same eval again: (1) and (4) are about hook/agent *configuration* the eval harness doesn't inspect, (2)/(3) are a state-persistence bug that the single-scenario e2e run happened not to trigger (subagents preserved the roster by good judgment, not because the instructions required it), and (5) is pure doc-consistency. Different failure classes need different-shaped checks, not more repetitions of the same one.
- **"All seats reported" is not the same as "the checkpoint was recorded" — orchestrating commands must verify their own writes.** The full-pipeline e2e eval's build stage had all 5 Boardroom seats genuinely return verdicts, yet the orchestrating agent never actually appended to `.wingman/checkpoints.jsonl` or updated `state.json` — it apparently considered the stage done once the seats replied, without confirming the subsequent file-write step actually executed. `boardroom.md`'s "Record the checkpoint" section already said "always, regardless of scope," so this wasn't a wording gap in what to do — it was a missing check that the doing had happened. Fixed by adding an explicit instruction to re-read both files from disk before reporting a checkpoint complete, naming the `verification-before-completion` skill directly. Worth generalizing: any command with a multi-step "do the work, then persist a record of it" shape is exposed to this same failure mode, and the fix is always the same — verify the persistence step landed, don't infer it from the preceding step having gone well.
- **A recurring-friction detector must distinguish "same incident, logged twice" from "genuine repetition."** `evolve-promotion`'s negative-case eval deliberately fed it three topics each appearing in exactly 2 files (a learning entry plus that same event's checkpoint or retro) to check it wouldn't naively count file-mentions as occurrences. It correctly recognized same-date, same-fix pairs as one incident each and proposed nothing — worth remembering as the specific failure mode this class of mechanism is most exposed to, since every piece of real work naturally produces a learning note *and* a checkpoint/retro for the same event.
- **`docs/` isn't part of the installed plugin — anything a skill needs at runtime must be bundled inside `plugins/wingman/`.** Caught during v3 work: `marketplace.json`'s `pluginRoot`/`source` only installs `plugins/wingman/` into a founder's Claude Code environment; the repo-root `docs/` folder (including `docs/AGENT-ROSTER.md`, which `evolve-promotion` needs to name specialists from) never ships. Fixed by adding `plugins/wingman/skills/evolve-promotion/references/specialist-catalog.md` as the runtime-shipped copy, with an explicit sync note in both files rather than silently duplicating content. Also removed a similar latent dependency from `department-lead-activation`'s template, which didn't actually need the specialist-level catalog at all (department leads operate at a coarser granularity). Any future skill referencing another doc for *operational* purposes (not just rationale/citation) needs this same check.
- **Evolve-promoted commands and skills follow the same founder's-project rule as specialist agents — not a plugin-directory exception.** The original `evolve.md`/`AGENT-ROSTER.md` text (written before `department-lead-activation` established the project-scoping precedent) said to write promoted commands/skills into `plugins/wingman/` while specialists went to the founder's project — an inconsistent carve-out. Fixed: `/wingman:evolve` never writes into `plugins/wingman/` for any artifact type; commands go to `.claude/commands/`, skills to `.claude/skills/`, agents to `.claude/agents/`, all in the founder's own project. "Wingman's plugin directory" always means the founder's resyncable local install, regardless of what's being written.
- **Retros need one canonical file, like `LEARNINGS.md` already has.** `retro.md` originally didn't specify where retro blocks get saved, leaving them scattered across whichever file happened to be open — unfindable by `/wingman:evolve`'s clustering step. Fixed: retros now append to `docs/wingman/retros.md`, mirroring `LEARNINGS.md`'s convention exactly.
- **Behavioral claims about a skill need a real, independently-verified run, not a worked example.** The department-lead-activation skill was initially "proven" only by a hand-written worked example showing what the template *would* produce. That's evidence the template is well-formed, not evidence the skill's instructions actually produce that outcome when a fresh agent follows them. `evals/` exists to close that gap — see `evals/README.md` and `evals/cases/department-lead-activation.md`'s run log for the actual verified run.
- **Department-lead files live in the founder's project, not in Wingman's plugin directory.** Considered writing dynamically-created `dept-*.md` agents into `plugins/wingman/agents/` at runtime; rejected because plugin files are resynced from the marketplace source and a mid-session write there risks silent loss on the next update, plus there's no guaranteed way to make a freshly-written *plugin* agent discoverable within the same session without a reload. Claude Code's project-scoped subagent mechanism (`.claude/agents/*.md`, auto-discovered, no manifest) is the correct home — it also matches intent, since each founder's project should accumulate its own department-lead roster, not share Wingman's.
- **Runtime provider: Claude Code native only.** Evaluated LangGraph and smolagents for the literal 56-agent blueprint's orchestration layer; rejected both because either would require a separate hosted Python service, turning Wingman into "a plugin + a product" instead of just a plugin. Cross-agent collaboration uses Claude Code's own Task-tool dispatch (built) and, where useful later, the experimental Agent Teams feature (not yet used). Persistent state, where needed, is a small local MCP server following `gsd-plugin`'s proven pattern — not a database, not LangGraph state.
- **Hybrid agent population, not the literal 56-agent roster upfront.** The full corporate-hierarchy blueprint is kept as a naming/organizing scheme (`docs/AGENT-ROSTER.md`), but only 5 Boardroom seats are built at install time; department leads and specialists grow lazily per-project. Rationale: a fixed 56-agent roster taxes every project's context budget regardless of whether that project ever needs a Row-Level-Security specialist or a canary-rollback agent.
- **Vendoring means "read and adapt," not "depend on."** All 16 vendor repos are git submodules for reference/attribution only. None of their bespoke infrastructure (SDKs, CLIs, MCP servers, hosted dashboards) is a runtime dependency of the installed Wingman plugin.
- **Boardroom seats never call other agents.** Adopted verbatim from `addyosmani-agent-skills`' persona/command distinction: only commands (`boardroom.md`, the pipeline stages) orchestrate and merge; agents (personas) only ever report back.
- **Checkpoint enforcement is a deterministic hook, not an LLM judgment call.** The `boardroom-checkpoint` hook checks for a marker string, not the plan's actual content — the LLM reasoning happens earlier, in `/wingman:boardroom` itself. This keeps the safety backstop auditable and testable independent of model behavior.
