## Decisions log

Durable decisions only — not every turn-level choice. Newest first.

- **Regression-hardening: mechanized the audit-found gaps into a two-layer hybrid, and drew a deliberate line at what NOT to mechanize.** The multi-expert audits kept finding bugs that were mechanically checkable but unchecked (inert hook, model-tier drift, doc-drift, orphan files, attribution error) — so they recurred silently until an expensive audit caught them. Layer 1 (cheap, deterministic, run every change) now catches them: `validate-structure.mjs` gained orphan detection, the model-tier invariant, and a skill-anatomy warning; a new `scripts/check-repo-consistency.mjs` catches command doc-drift and attribution coverage. Every check was verified both ways — real repo passes clean with zero false positives, and reintroducing each historical bug fires the check. Crucially, two candidates were deliberately **left un-mechanized** (attribution license-accuracy, non-shipped-doc runtime deps): a keyword heuristic false-positives on its own corrective text — the accurate fix for the andrej-karpathy error literally contains the words "no license" while explaining the old mistake — and a noisy check trains people to ignore warnings, worse than nothing (the `systematic-auditing` skill's own reasoning). Those are Layer-2 items in the new `docs/REGRESSION-CHECKLIST.md`, run via `/wingman:audit`. General lesson recorded: the honest enterprise-grade move is mechanize-what's-crisp + checklist-what's-semantic, not mechanize-everything.
- **Genuine evolve gather on this session's own signal correctly proposed nothing new — because the two real recurring patterns were already promoted.** Ran the `/wingman:evolve` gather/cluster logic against the 19-entry decisions log. Two patterns genuinely recurred 3+ times: (1) a confidently-stated claim repeated across docs but never re-verified against the primary source, turning out wrong (the `PermissionRequest` hook name, the andrej-karpathy license claim, a non-shipped-`docs/` runtime dependency); (2) a multi-step "do work, then persist a record" that skipped or clobbered the persist step (the dropped build checkpoint, the `state.json`-clobbering overwrite, `department-lead-activation` never writing state). Both cleared the 2+ threshold — and both had *already* been acted on this session (pattern 1 → the `systematic-auditing` skill + the mechanized validators; pattern 2 → `boardroom.md`'s read-merge-write and re-read-from-disk fixes). So the correct output was to propose no new specialist/skill/command: a fresh one would duplicate what `systematic-auditing`/`verification-before-completion` already cover, which `engineering-minimalism` forbids. This is evolve working correctly — recognizing genuine repetition *and* recognizing when it's already resolved — not evolve finding nothing.

- **Merged to `main` on explicit request, after a dedicated pre-merge regression check rather than merging on the strength of prior evidence alone.** Before merging PR #1, folded in PR #2 (the hook-event-name validation fix from the PR review) first, since merging without a verified fix sitting right there would have left known risk-reduction on the table — flagged this deviation from the literal "merge PR #1" instruction transparently and got explicit confirmation before continuing, since a merge with no separate human review of its own is exactly the kind of unilateral action worth pausing on. Then, before merging: re-ran `validate-structure.mjs` fresh on the merged branch (PASS); directly re-simulated the original hook bug one more time to confirm the new check still catches it; and checked chronologically that every behavioral file changed since its own last eval verification had already been re-exercised by a later real pipeline run (Phase 3's Run 2 postdates every fix to `boardroom.md`/`secure.md`/`department-lead-activation`) — so nothing merged represented untested drift. Did not do a full re-run of all 10 eval cases, since nothing they cover had changed since their last verified run; re-running unchanged, already-verified behavior would have cost real tokens for zero new information. `main` now has everything this project has built.

- **A human-requested sweep of "what else needs a human" (env vars, API keys, screenshots) found a real, previously-repeated attribution error, not just new to-do categories.** Checked properly this time (the original `docs/HUMAN-TODOS.md` only carried over 2 items already tracked elsewhere) and confirmed Wingman needs zero credentials/env vars, but found: `main` on GitHub has never received any of this project's work (everything lives on `claude/init-6eg3d1` only); zero demo/screenshot content exists anywhere, and can't be produced from this sandbox honestly (every eval ran against throwaway fixtures with simulated `AskUserQuestion` answers); and, doing a self-check pass on `ATTRIBUTIONS.md` before flagging it for human/legal review, found that `andrej-karpathy-skills` had been documented in three separate places (`ATTRIBUTIONS.md`, `ARCHITECTURE.md` §9, and `engineering-minimalism/SKILL.md`'s own header comment) as having "no LICENSE file," when it actually declares MIT in three places (`plugin.json`, `README.md`, `SKILL.md` frontmatter) — just not as a standalone `LICENSE` file. Fixed everywhere. No real exposure resulted (the content was always restated in Wingman's own words, never quoted), but it's a concrete instance of exactly the failure mode `systematic-auditing` exists to catch — a claim repeated confidently across multiple docs, never independently re-checked against the actual source. Worth remembering: attribution/licensing claims are exactly the kind of thing that should get the same "re-verify against the primary source, don't just trust the existing doc" treatment as any other audit finding.

- **Prioritized eval coverage by actual risk/value instead of working the untested-file list top to bottom.** After Phase 3, roughly a dozen commands/skills still had zero dedicated behavioral evidence. Rather than testing all of them, three were identified as genuinely high-value (`systematic-auditing`/`audit.md` — brand new, unproven, directly serves the session's own rigor-codification request; `verification-before-completion` — the single most load-bearing discipline skill in the project, everything else's "independently verify" methodology is this skill applied one level up, yet it had never itself been tested; `retro.md` — zero prior exercise, cheap to test using already-existing fixture history) and the rest deliberately deferred with stated reasoning (`design-taste` has no UI-having fixture in this project to test against without meaningfully more setup; `engineering-minimalism`/`plain-language-checkpoint` are exercised so pervasively and incidentally across nearly every other eval that a dedicated case would mostly restate existing evidence; `token-economy` governs internal-only channels with no founder-visible output to grade against). One real, unplanned finding from the `systematic-auditing` run: nested subagent dispatch (a `general-purpose` agent spawning further `general-purpose` agents) actually works in this sandbox — a more precise statement of this project's sandbox limits than the earlier-established "named custom `subagent_type`s don't resolve," which is still true but is a narrower constraint than "no nested dispatch at all."
- **Two full real-dispatch pipeline runs, zero genuine `NO_GO`s — treated as a real, informative result, not a shortfall to keep re-rolling for.** `full-pipeline-e2e` Run 2 (a mid-life "Ledger" fixture, deliberately seeded with a flawed hardcoded-API-key snippet and a repeatable rollback-migration mistake) completed all 4 stages with genuine, non-synthetic 5-seat Boardroom dispatch throughout, including a real fix-recheck loop, an independently-reconfirmed Cost-seat finding, and a self-caught, honestly-disclosed ship-stage process defect. Like Run 1, it never produced a `NO_GO`: both seeded problems were avoided by good engineering judgment (a keyless API chosen over the flawed snippet; the historical lesson correctly applied) rather than needing the Boardroom to catch them after the fact. Decision: don't chase a third "realistic" scenario hoping one eventually produces a `NO_GO` by chance — that's not really testing anything new, since `boardroom-gate-rule.md` already independently verifies the `DO NOT SHIP` file-writing/`next_stage`-pinning mechanics work correctly (with real file-write checks, via synthetic verdicts). What remains genuinely untested is specifically "real independent reviewers reject a real bad diff," which needs a fixture deliberately adversarial enough that competent engineering judgment plausibly doesn't route around it — a different kind of construction than "add realistic seeded flaws and see what happens," which has now been tried twice with the same (informative) result both times.

A real founder using the installed plugin won't know to use those exact phrases. Fixed by adding `skills/systematic-auditing` (paired naming with `systematic-debugging`) with those exact trigger phrases in its frontmatter `description` — per this project's own "description trap" lesson, a vague description means a skill never reliably auto-routes — plus `commands/audit.md` as an explicit, discoverable on-demand entry point that doesn't depend on the founder phrasing anything a particular way. Deliberately scoped as a cross-cutting process skill, not a 9th department — `docs/ARCHITECTURE.md` §2/§3 has a real, considered principle against extending the fixed 8-department roster ad hoc, and this doesn't fit that shape (no per-project activation signal, no remit over a body of work — it's a *how*, like `systematic-debugging`, not a *what*). Same session, also added `docs/wingman/founder-todos.md` as a per-project convention (wired into `secure.md`'s existing founder-accepted-risk step): founder-only business-risk decisions were previously only recorded in a checkpoint's `founder_notes` field, unscannable without reading the whole audit log — now they land in one dedicated list, mirroring `retros.md`'s established precedent rather than inventing a new storage convention.

Findings: (1) `next_stage`'s meaning on a `DO NOT SHIP` checkpoint was undefined — since `state.json`'s `current_stage` is set from `next_stage` unconditionally, a literal reading of the schema's other (advancing) examples could have let a blocked checkpoint silently advance the project past its own gate; now explicit that `next_stage` stays pinned to the reviewed stage on `DO NOT SHIP`. Neither gap could have been found by re-running `full-pipeline-e2e` again, since that eval's single scenario never produced a `DO NOT SHIP` outcome or exercised the directly-passed-content path more than once. Token-efficiency choice made explicitly: declined to build a separate dedicated eval for the 5 Boardroom personas, since they'd already been dispatched ~20+ times incidentally across other evals with their output contract holding every time — spending a fresh eval on them would mostly re-confirm existing evidence.
- **A four-way parallel expert audit (architecture, content quality, executable code, eval coverage) found and fixed 5 real bugs no single pass had caught.** Run 2026-07-07 as a deliberate cross-check after the e2e eval, each auditor scoped to a distinct concern and given exact file lists to stay token-efficient. Different failure classes need different-shaped checks, not more repetitions of the same one.
- **"All seats reported" is not the same as "the checkpoint was recorded" — orchestrating commands must verify their own writes.** The full-pipeline e2e eval's build stage had all 5 Boardroom seats genuinely return verdicts, yet the orchestrating agent never actually appended to `.wingman/checkpoints.jsonl` or updated `state.json` — it apparently considered the stage done once the seats replied, without confirming the subsequent file-write step actually executed. `boardroom.md`'s "Record the checkpoint" section already said "always, regardless of scope," so this wasn't a wording gap in what to do — it was a missing check that the doing had happened. Fixed by adding an explicit instruction to re-read both files from disk before reporting a checkpoint complete, naming the `verification-before-completion` skill directly. Worth generalizing: any command with a multi-step "do the work, then persist a record of it" shape is exposed to this same failure mode, and the fix is always the same — verify the persistence step landed, don't infer it from the preceding step having gone well.
- **A recurring-friction detector must distinguish "same incident, logged twice" from "genuine repetition."** `evolve-promotion`'s negative-case eval deliberately fed it three topics each appearing in exactly 2 files (a learning entry plus that same event's checkpoint or retro) to check it wouldn't naively count file-mentions as occurrences. It correctly recognized same-date, same-fix pairs as one incident each and proposed nothing — worth remembering as the specific failure mode this class of mechanism is most exposed to, since every piece of real work naturally produces a learning note *and* a checkpoint/retro for the same event.
- **`docs/` isn't part of the installed plugin — anything a skill needs at runtime must be bundled inside `plugins/wingman/`.** Caught during v3 work: `marketplace.json`'s `pluginRoot`/`source` only installs `plugins/wingman/` into a founder's Claude Code environment; the repo-root `docs/` folder (including `docs/AGENT-ROSTER.md`, which `evolve-promotion` needs to name specialists from) never ships. Fixed by adding `plugins/wingman/skills/evolve-promotion/references/specialist-catalog.md` as the runtime-shipped copy, with an explicit sync note in both files rather than silently duplicating content. Also removed a similar latent dependency from `department-lead-activation`'s template, which didn't actually need the specialist-level catalog at all (department leads operate at a coarser granularity). Any future skill referencing another doc for *operational* purposes (not just rationale/citation) needs this same check.
- **Evolve-promoted commands and skills follow the same founder's-project rule as specialist agents — not a plugin-directory exception.** The original `evolve.md`/`AGENT-ROSTER.md` text (written before `department-lead-activation` established the project-scoping precedent) said to write promoted commands/skills into `plugins/wingman/` while specialists went to the founder's project — an inconsistent carve-out. Fixed: `/wingman:evolve` never writes into `plugins/wingman/` for any artifact type; commands go to `.claude/commands/`, skills to `.claude/skills/`, agents to `.claude/agents/`, all in the founder's own project. "Wingman's plugin directory" always means the founder's resyncable local install, regardless of what's being written.
- **Retros need one canonical file, like `LEARNINGS.md` already has.** `retro.md` originally didn't specify where retro blocks get saved, leaving them scattered across whichever file happened to be open — unfindable by `/wingman:evolve`'s clustering step. Fixed: retros now append to `docs/wingman/retros.md`, mirroring `LEARNINGS.md`'s convention exactly.
- **Behavioral claims about a skill need a real, independently-verified run, not a worked example.** The department-lead-activation skill was initially "proven" only by a hand-written worked example showing what the template *would* produce. That's evidence the template is well-formed, not evidence the skill's instructions actually produce that outcome when a fresh agent follows them. `evals/` exists to close that gap — see `evals/README.md` and `evals/cases/department-lead-activation.md`'s run log for the actual verified run.
- **Department-lead files live in the founder's project, not in Wingman's plugin directory.** Considered writing dynamically-created `dept-*.md` agents into `plugins/wingman/agents/` at runtime; rejected because plugin files are resynced from the marketplace source and a mid-session write there risks silent loss on the next update, plus there's no guaranteed way to make a freshly-written *plugin* agent discoverable within the same session without a reload. Claude Code's project-scoped subagent mechanism (`.claude/agents/*.md`, auto-discovered, no manifest) is the correct home — it also matches intent, since each founder's project should accumulate its own department-lead roster, not share Wingman's.
- **Runtime provider: Claude Code native only.** Evaluated LangGraph and smolagents for the literal 56-agent blueprint's orchestration layer; rejected both because either would require a separate hosted Python service, turning Wingman into "a plugin + a product" instead of just a plugin. Cross-agent collaboration uses Claude Code's own Task-tool dispatch (built) and, where useful later, the experimental Agent Teams feature (not yet used). Persistent state, where needed, is a small local MCP server following `gsd-plugin`'s proven pattern — not a database, not LangGraph state.
- **Hybrid agent population, not the literal 56-agent roster upfront.** The full corporate-hierarchy blueprint is kept as a naming/organizing scheme (`docs/AGENT-ROSTER.md`), but only 5 Boardroom seats are built at install time; department leads and specialists grow lazily per-project. Rationale: a fixed 56-agent roster taxes every project's context budget regardless of whether that project ever needs a Row-Level-Security specialist or a canary-rollback agent.
- **Vendoring means "read and adapt," not "depend on."** All 16 vendor repos are git submodules for reference/attribution only. None of their bespoke infrastructure (SDKs, CLIs, MCP servers, hosted dashboards) is a runtime dependency of the installed Wingman plugin.
- **Boardroom seats never call other agents.** Adopted verbatim from `addyosmani-agent-skills`' persona/command distinction: only commands (`boardroom.md`, the pipeline stages) orchestrate and merge; agents (personas) only ever report back.
- **Checkpoint enforcement is a deterministic hook, not an LLM judgment call.** The `boardroom-checkpoint` hook checks for a marker string, not the plan's actual content — the LLM reasoning happens earlier, in `/wingman:boardroom` itself. This keeps the safety backstop auditable and testable independent of model behavior.
